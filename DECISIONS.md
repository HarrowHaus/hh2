# DECISIONS.md
Log every non-obvious build decision here, newest first. Format: date · decision · why.

- **2026-07-02 · Browser: guestbook CUT; webring DEMOTED from the default start page (docs/14).** Owner ruling, folded into the new `docs/14-browser-bookmarks.md` (which supersedes the browser-content scattered across docs/07/08). Two reconciliations: (1) the Underground Noise Webring is no longer the homepage — the Browser now opens on a plain, period-plausible personal portal (`about:start` → `PortalPage`: a search box + a short personal links list in the owner's voice, no feature-announcing menus, no meta-narrative per Rule 2); the webring stays as content and remains reachable as a bookmark ("the ring") / by digging, per docs/14 §1. (2) The **guestbook is cut** — do not build it; ignore any earlier guestbook spec (docs/03, docs/05, PHASE3-content-manifest). Why: discovered-not-presented (Rule 2) — the start page should not announce a feature. Scope note: the curated §3/§4 bookmark set is **not** built yet (deferred per the task); and the guestbook UI still rendered on the legacy `moldmouth's corner` in-world page (`HomePage`) is left in place for now — the start page no longer routes there, but the dead guestbook markup should be removed in a follow-up pass to fully honor the cut.

- **2026-06-29 · Added `LICENSE` (MIT © 2026 Harrowhaus / Donald Pilger) + `CREDITS.md`, fulfilling the Phase-0 attribution commitment.** A state-of-parity audit confirmed the build is an **original reimplementation** — no upstream source is vendored (deps = React/React-DOM/Zustand only; all assets original SVG), so `ShizukuIchi/winXP` (MIT © 2019 Shizuku Yang) is credited as the XP-chrome/reducer **reference** and `DustinBrett/daedalOS` (MIT © 2025 Dustin Brett) as the **behavioral/parity** reference, with both MIT texts reproduced in `CREDITS.md`. This is credit, not redistribution. CREDITS.md is the running ledger for third-party apps adopted in later tiers (`docs/08-app-roadmap.md`) — each logged with license + branding terms; GPL/AGPL/non-commercial/name-restricted ones flagged as owner decisions before adoption.

- **2026-06-29 · Phase 2 WM parity: virtual FS persisted to IndexedDB; window/session state persisted to localStorage.** The FS (node map + desktop-icon layout) uses a tiny IndexedDB `StateStorage` adapter (`src/os/fs/idb.ts`) per docs/02 #7/#10 — room to grow beyond the ~5MB localStorage bucket. Lighter session state (visual style + open windows) stays in localStorage via zustand `persist`. Transient flags (boot/login/start-menu/snap-preview) are never persisted, so each load re-boots → re-logs-in → restores windows. Window snapping follows the daedalOS parity target (top→maximize, side→half-tile) rather than literal XP, since daedalOS is the acceptance bar (CLAUDE.md Rule 1).

- **2026-06-29 · Theming engine refactored to pack/manifest-driven + `border-image`-ready (docs/02, docs/06).** The 3 styles were hard-coded as `body.theme-*` CSS blocks toggled by class; per the updated briefs they are now `ThemePack` manifests (`src/os/theme/packs.ts`, schema = docs/06) that the engine (`theme/engine.ts`) applies to `:root` at runtime. `tokens/themes.css` keeps only the dark values as the no-JS first-paint default + the variable contract. Chrome carries inert `border-image` hooks (window caption, control buttons, taskbar, Start — width 0 → no visual change) so a future pack (incl. a converted real `.msstyles`) lights up the 9-slice with zero engine/markup changes.
  Why: makes the deferred real-`.msstyles` loader (docs/06, Phase 5) an additive module, not a rewrite. docs/06 + Phase 5 stay deferred — only the open architecture is built now. Verified: all 3 styles render identically to the hard-coded version; engine sets `data-theme` + custom props on `:root`.

- **2026-06-29 · Base repo = `ShizukuIchi/winXP` for XP chrome/feel; `DustinBrett/daedalOS` kept as the FS/WM parity reference. Rebuilt on the mandated modern stack (Vite + React + TS + Zustand + CSS-modules + theme-token custom properties) rather than forking either build wholesale.**
  Why:
  - **Aesthetic is locked to authentic XP chrome (CLAUDE.md Rule 4).** `winXP` is the most period-accurate XP recreation in the wild — its taskbar, two-panel Start menu, title bars and window controls already *are* the look we want. daedalOS's default shell is not XP; re-skinning its large Next.js 15 / React 19 / BrowserFS architecture (80+ deps) down to XP would mean fighting the grain of a complex codebase and risks trading function for looks — the inverse of Rule 1. So we deepen winXP toward daedalOS parity (the CLAUDE.md Phase-0 default), not the reverse.
  - **daedalOS stays the parity reference** (docs/02). It is far deeper on virtual FS + window manager; we pull function toward it ticket-by-ticket and audit side-by-side, but we do not adopt its chrome.
  - **Stack mismatch → rebuild, don't fork-as-is.** winXP is CRA + styled-components + JS + React 16, with no state lib and no TypeScript. The brief mandates Vite + React + TS + Zustand + CSS-modules + theme tokens (so all three visual styles derive from `tokens/themes.css` and future styles drop in). We therefore scaffold the mandated stack now and **port winXP's authentic chrome onto it in Phase 1** ("build XP chrome from the winXP base; only the skin is ours"), instead of inheriting CRA/styled-components.
  - **Licensing — both MIT, compatible.** `ShizukuIchi/winXP`: MIT, © 2019 Shizuku Yang (LICENSE file present; `package.json` omits the field). `DustinBrett/daedalOS`: MIT. We keep XP-derived assets as **original recreations**, not lifted Microsoft bitmaps (docs/04), and will carry winXP's MIT attribution where its chrome/markup is ported.
  - Note: winXP also bundles `webamp` — relevant later to the foobar2000 / music pillar, though our player is custom per docs/03.
